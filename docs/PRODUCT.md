# 产品需求文档

## 1. 产品定义

Phoenix（工作名称）是一个面向网站开发的 Rust 全栈框架。它为喜欢 Laravel 工作流、但希望采用 Rust 的开发者提供一条低认知负担的路径：业务代码围绕路由、控制器、请求对象、模型和 React 页面组织，而不是直接拼装底层 HTTP、ORM 和前端构建工具。

产品不是 Laravel 的逐行移植。目标是复现 Laravel 的清晰性、一致性和开发效率，同时遵守 Rust 的所有权、类型与异步模型。

## 2. 目标用户

### 主要用户

- 熟悉 PHP/Laravel，第一次或刚开始使用 Rust 的网站开发者。
- 希望用一个仓库开发 Rust 后端与 React 前端的小型团队。
- 需要服务端路由和业务控制，同时不想单独维护 REST API 契约的产品团队。

### 次要用户

- 需要类型安全、性能和可部署单体应用的 Rust 开发者。
- 需要逐步从服务端页面迁移到 React 交互页面的项目。

## 3. 用户问题

现有 Rust Web 生态组件质量高，但新手通常需要自行选择并连接运行时、HTTP 框架、路由、ORM、验证、会话、CSRF、前端构建、错误处理和测试工具。每个组件的 API 风格与配置方式不同，导致开始一个普通网站项目也需要大量架构决策。

Phoenix 要解决的不是“缺少 HTTP 服务器”，而是“缺少一套连贯、默认可用、文档一致的网站开发产品”。

## 4. 产品原则

1. **约定优于配置**：提供可靠默认目录、错误页、数据库与前端连接方式。
2. **显式优于魔法**：便利 API 必须能追踪到具体类型与执行路径，编译错误应尽量可理解。
3. **渐进式深入**：新手可以使用默认方案，熟练用户可以替换底层组件或进入更低层 API。
4. **一个请求，一条清晰链路**：路由到请求验证、控制器、模型、React props 与响应应可被日志和测试完整观察。
5. **安全默认开启**：TLS 部署指引、CSRF、会话 Cookie、安全响应头、敏感字段过滤与错误脱敏默认生效。
6. **稳定门面**：应用代码优先依赖 Phoenix API，底层 Hyper、Toasty 或 Vite 的升级由适配层吸收。
7. **契约只定义一次**：请求字段、页面 Props 与 Resource 在 Rust 中声明，React 自动获得对应类型和运行时字段描述。

## 5. 核心用户旅程

### 5.1 展示列表页

1. 开发者在 `routes/web.rs` 注册命名路由。
2. 控制器调用模型查询数据。
3. 控制器通过 `render("posts/index", props)` 返回页面。
4. 框架解析 Vite 资源清单，并把页面名、props、共享数据和安全元数据交给 React 启动器。
5. `views/pages/posts/index.tsx` 接收类型化 props 并渲染。

### 5.2 提交并验证表单

1. React 表单携带 CSRF token 提交普通表单或 JSON 请求。
2. 请求对象完成字段提取、规范化、验证和授权检查。
3. 验证失败时，框架返回结构化错误；浏览器导航请求可自动回跳并恢复安全的旧输入。
4. 验证成功时，控制器收到已验证类型，而不是未约束的字符串映射。
5. 控制器保存模型，写入一次性消息并重定向到命名路由。

### 5.3 数据库变更

1. 开发者运行 `phoenix make:migration` 或 `phoenix make:model Post --migration` 生成带稳定 ID、`up` 和 `down` 的迁移骨架。
2. 迁移包含稳定 ID、描述、`up` 和在可逆情况下的 `down`。
3. 迁移执行器通过应用二进制或专用示例入口运行，记录已执行版本并使用数据库锁避免并发执行。
4. Toasty 模型与迁移 schema 的一致性在测试或开发启动检查中验证。

### 5.4 创建登录表单

1. 开发者在 Rust `LoginRequest` 中定义 `user`、`password` 和验证规则。
2. 构建流程自动生成带命名空间的 `LoginInput` TypeScript 类型与表单契约。
3. React 通过类型安全的 `form.field("user")` 和 `form.field("password")` 绑定控件，不再重写字段接口。
4. Rust 字段、Serde wire name 或验证规则变化后，前端类型与运行时描述自动刷新。
5. 同名契约或最终 wire name 冲突时构建失败，并报告冲突来源。

### 5.5 选择页面渲染方式

1. 应用选择默认 React 渲染模式。
2. 内容页可以使用 SSR，后台可以使用 SPA，文档页可以使用 Islands。
3. 控制器、Props、验证错误和路由保持不变，渲染层负责不同的 HTML 与 hydration 策略。
4. 构建产物记录资源版本与契约 hash，防止 Rust 服务和 React renderer 使用不一致代码。

## 6. 功能范围

### P0：首个可用版本必须具备

| 模块 | 用户能力 | 最低验收标准 |
| --- | --- | --- |
| 应用核心 | 配置、状态、环境、错误与生命周期 | 示例应用可启动、优雅关闭并输出结构化日志 |
| 路由 | HTTP 方法、参数、命名路由、分组、中间件 | 静态/动态路由匹配正确，可从名称生成 URL |
| 控制器 | 异步处理器、依赖提取、统一响应 | 可返回页面、JSON、重定向、文件和标准错误 |
| 请求 | Query、Path、Header、Form、JSON、Multipart 基础支持 | 非法内容类型与超限载荷返回明确错误 |
| 验证 | 规则、嵌套字段、自定义规则、错误消息、已验证 DTO | 失败错误稳定且字段级可序列化 |
| 数据契约 | 从 Rust Request、Props、Resource 自动生成 TypeScript 类型和运行时字段描述 | 登录表单无需重复定义字段，冲突与陈旧契约会使构建失败 |
| React SPA | `.tsx` / `.jsx` 页面发现、props、共享数据、Vite 开发与生产资源 | 刷新、局部跳转、验证错误和生产构建均可用 |
| 模型 | Toasty 模型门面、CRUD、查询、关系、分页、事务 | SQLite 与 PostgreSQL 至少通过契约测试 |
| 迁移 | 创建/删除/修改表的迁移 API、状态表、执行与回滚 | 可在空库升级并在测试库回滚 |
| 中间件 | 请求 ID、日志、错误捕获、会话、CSRF、安全头 | 默认 Web 栈通过安全集成测试 |
| 配置 | 环境变量、开发/测试/生产配置、密钥校验 | 生产环境缺少关键配置时拒绝启动 |
| 测试 | 请求测试、数据库隔离、页面 props 断言 | 示例应用关键旅程可自动验证 |
| CLI | 新建项目与业务代码生成 | 新项目可直接编译运行，模型全量生成能自动连接迁移、Request、Resource、控制器、路由与页面 |

### P1：首个稳定版本

- 认证基础设施：登录、登出、密码哈希、认证中间件与用户接口，但不强绑定 UI。
- 文件存储抽象与本地驱动。
- 本地化验证消息。
- 限流、可信代理和 CORS 配置。
- 数据库 seed/factory 基础能力。
- React 页面懒加载、错误边界与代码分割。
- React SSR：持久 renderer、完整 HTML、hydration、流式响应、超时和故障处理。
- React Islands：服务端 HTML、独立 island 入口和选择性 hydration。

### 暂不进入首版

- 队列、定时任务、邮件、通知、广播与 WebSocket。
- Blade 模板兼容层。
- Laravel 包兼容或 PHP 代码迁移工具。
- 管理后台、CMS、低代码页面编辑器。
- 多租户、微服务编排或分布式事务。

## 7. React 页面协议

服务端与浏览器之间使用版本化页面协议，而不是把任意 Rust 对象直接注入脚本：

```json
{
  "protocol": 1,
  "render_mode": "spa",
  "page": "users/show",
  "props": {},
  "shared": {},
  "errors": {},
  "flash": {},
  "request_id": "...",
  "contract_hash": "...",
  "asset_version": "..."
}
```

协议必须满足：

- 所有 props 通过 `serde::Serialize` 进入结构化序列化流程。
- HTML 初始载荷使用安全转义，禁止通过字符串拼接生成 `<script>` 内容。
- 页面名采用受约束的逻辑路径，不能访问 `views/` 外文件。
- 共享数据按请求惰性求值，避免每次导航加载昂贵数据。
- 协议包含版本与资源版本，客户端检测不兼容时执行完整刷新。
- Rust/TypeScript 契约 hash 必须匹配；不匹配时 SPA 完整刷新，SSR/Islands renderer 拒绝使用陈旧 bundle。

## 8. 数据传输与“加密”定义

“加密传输”拆成三个不同问题：

| 目标 | 方案 | 产品承诺 |
| --- | --- | --- |
| 防止链路窃听与篡改 | 生产环境强制 HTTPS/TLS，正确处理可信代理 | P0 必须 |
| 保护 Cookie 与服务端状态 | `Secure`、`HttpOnly`、`SameSite` Cookie；会话 ID 随机且可轮换；敏感状态留在服务端 | P0 必须 |
| 对特定 props 做应用层封装 | 可选的 AEAD 加密或签名信封、短时效、用途绑定和防重放 | P1 候选 |

浏览器若要渲染某个值，最终一定能读取明文，因此应用层加密不能对浏览器用户保密。密码、令牌、内部字段等不应发送到 React；模型序列化必须采用显式 ViewModel/Resource 白名单，不能默认暴露全部字段。

## 9. 成功指标

首个可用版本以可验证结果而不是功能数量衡量：

- 新项目从克隆示例到看到数据库驱动的 React 页面不超过 10 分钟。
- 常规页面开发只需修改路由、请求对象、控制器、模型和 `views/` 文件。
- 前端不手写与 Rust 请求或页面 Props 重复的字段接口。
- 示例应用覆盖列表、详情、创建、验证失败、更新、删除和登录态保护。
- P0 公共 API 有文档示例与集成测试。
- 稳定版前不保留未说明的 `unsafe`，依赖漏洞扫描无已知高危问题。

## 10. 发布门槛

### 技术预览

- SQLite 示例应用端到端可运行。
- Rust 契约到 TypeScript 的自动生成、React SPA 开发服务器与生产资源构建均验证。
- API 可破坏性变更被允许，但必须记录。

### Alpha

- PostgreSQL 契约测试通过。
- 迁移、会话、CSRF 与验证错误链路完整。
- SSR renderer、完整 HTML 和 hydration 链路通过集成测试。
- 关键模块有公共文档和错误诊断。

### Beta

- 完成安全审查、性能基线和升级指南。
- Islands 选择性 hydration 和客户端 bundle 预算通过验证。
- 公共 API 进入变更控制，示例项目能够完成一次版本升级。

### 1.0

- P0/P1 承诺明确，长期支持范围与最低 Rust 版本写入政策。
- 至少一个非示例应用完成试用反馈闭环。
